    ```java
      /**
       * Domain entry point for the FE checkout flow. Derives totalAmount from items, defaults status
       * to PENDING, persists via the existing OrderEntity factory. The userId is supplied by the
       * controller after reading the X-User-Id session header — the service treats it as already
       * trusted (Phase 5: replace header trust with JWT claim verification at the gateway).
       */
      public OrderEntity createOrderFromCommand(String userId, CreateOrderCommand command) {
        if (userId == null || userId.isBlank()) {
          throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Missing X-User-Id session header");
        }

        BigDecimal totalAmount = command.items().stream()
            .map(item -> item.unitPrice().multiply(BigDecimal.valueOf(item.quantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);

        String note = command.note();   // null is fine — OrderEntity.note is nullable

        OrderEntity order = OrderEntity.create(userId, totalAmount, "PENDING", note);
        return repository.saveOrder(order);
      }
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /api/orders/orders body | Untrusted input; `@Valid` triggers existing handleValidation branch; nested validation on items[] catches invalid productId/quantity/unitPrice |
| `X-User-Id` header → service `userId` | TRUSTED via header for MVP. Phase 5 hardening: gateway verifies JWT and forwards a verified-userId header |
| client unitPrice → server-side totalAmount | Client supplies price snapshot; backend computes total from it. Mismatch with authoritative product-service price is allowed for MVP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05-01 | Spoofing | X-User-Id header trust | accept | MVP scope. The FE bearer-token flow already runs through middleware + http.ts; gateway-side JWT verification is a Phase 5 follow-up. Documented in this plan's `<interfaces>` and SUMMARY's "Deferred" section. Threat acknowledged. |
| T-04-05-02 | Tampering | Client-supplied unitPrice | accept | Client could send a discounted unitPrice and have the backend persist a low totalAmount. For the MVP (no payment-service integration in checkout chain) this is acceptable because no money actually moves. Phase 5 follow-up: re-fetch authoritative product price server-side before computing totalAmount, or move totalAmount computation entirely to a downstream order-fulfillment service. |
| T-04-05-03 | Information Disclosure | Validation error rejected-value leak | mitigate | Existing Phase 03 `sanitizeRejectedValue` in order-service `GlobalExceptionHandler` masks tokens/secrets/passwords. Plan does not change that behavior; new field validations (productId/items.unitPrice) flow through the same masking pipeline. |
| T-04-05-04 | Denial of Service | items[] unbounded list | accept | `@NotEmpty` validates non-empty but does not cap the list size.
